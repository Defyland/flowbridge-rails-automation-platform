# FlowBridge Event Contracts

FlowBridge events describe the timeline around workflow publication, webhook ingress, execution state, node attempts, retries, and dead letters. The MVP stores the source of truth in relational tables, while these events define the vocabulary for audit timelines, integrations, and future external consumers.

Workflow execution always references an immutable workflow version. Events must never imply that a mutable workflow draft was executed.

## Envelope

Every event must include:

- `event_id`: globally unique event identifier generated by FlowBridge, unless explicitly mapped from an external webhook event.
- `event_type`: stable dot-delimited event name.
- `schema_version`: integer schema version.
- `occurred_at`: UTC timestamp when the fact became true.
- `producer`: `flowbridge` for internal events or the source identifier for external webhook facts.
- `organization_id`: tenant boundary.
- `workflow_id`: mutable workflow container.
- `workflow_version_id`: immutable executable artifact.
- `execution_id`: workflow execution when one exists.
- `source`: provider or subsystem that produced the event, for example `stripe`, `github`, `operator_console`, or `flowbridge`.
- `source_event_id`: external source event identifier when present.
- `correlation_id`: request or webhook correlation key.
- `payload`: masked, schema-specific data.

## Idempotency policy

FlowBridge uses two related idempotency boundaries:

- Consumer deduplication uses `event_id`.
- Webhook ingress deduplication uses `source + source_event_id + workflow_version_id`.

For product discussions this is often shortened to `event_id/source/workflow`, but implementation must retain the immutable `workflow_version_id`. If a provider's `event_id` is only unique per account or topic, the `source` value must include that provider account/topic boundary.

Duplicate webhook delivery must not create a second `WorkflowExecution`. Duplicate internal event delivery must not cause consumers to apply the same state transition twice.

## Compatibility policy

- Additive payload fields are allowed within the same schema version.
- Removing or changing meaning of a field requires a new schema version.
- Execution and node events must reference the workflow version and graph checksum that actually ran.
- Secret-bearing values must be masked before entering event payloads.
- Dead-letter replay appends new attempt events instead of mutating old evidence.
- Consumers must tolerate unknown event types from newer producers by ignoring them or parking them for review.

## Event families

### Workflow events

Workflow events describe publication lifecycle, not execution.

- `workflow.version.published`
- `workflow.version.retired`

Required payload fields:

- `version_number`
- `graph_checksum`
- `trigger_key`
- `published_by`

### Webhook events

Webhook events describe ingress decisions before and after signature/idempotency checks.

- `webhook.event_received`
- `webhook.event_rejected`
- `webhook.event_duplicate`

Required payload fields:

- `source_event_id`
- `signature_algorithm`
- `trigger_key`
- `idempotency_key`
- `decision`

### Execution events

Execution events describe the workflow execution state machine.

- `workflow.execution.started`
- `workflow.execution.retrying`
- `workflow.execution.succeeded`
- `workflow.execution.failed`
- `workflow.execution.replay_requested`

Required payload fields:

- `status`
- `attempt`
- `retryable`
- `reason`

### Node events

Node events describe per-node evidence and are the primary debugging surface for operators.

- `workflow.node.started`
- `workflow.node.succeeded`
- `workflow.node.failed`
- `workflow.node.skipped`
- `workflow.node.retrying`

Required payload fields:

- `node_key`
- `node_type`
- `attempt`
- `duration_ms`
- `input`
- `output`
- `error`

`input`, `output`, and `error` must be masked before persistence or publication.

### Dead-letter events

Dead-letter events describe terminal failure and operator remediation.

- `dead_letter.created`
- `dead_letter.retry_requested`
- `dead_letter.resolved`

Required payload fields:

- `dead_letter_id`
- `reason`
- `resolved_by`
- `retry_execution_id`
